# Memory Update Skill

> Capture learnings from conversations and update CRM profiles

## Trigger
`/memory` or "update memory" or "save this to memory"

## What It Does
Scans the current conversation for:
1. **Person facts** - Things learned about specific people
2. **General facts** - Non-person learnings (optional: save to Core Memory)

Then:
- Fuzzy matches person names to `contacts.yaml`
- Updates/creates profile files in `Vault/4 - CRM/Profiles/{name}.md`
- Creates new contacts if person not found

## Execution Steps

### 1. Extract Facts from Conversation

Analyze the recent conversation and extract:

```typescript
interface ExtractedFacts {
  personFacts: { name: string; fact: string }[];
  generalFacts: string[];
}
```

**Person fact patterns to look for:**
- "Marcus works at a16z"
- "Talked to Sarah, she mentioned X"
- "Chris is leaving his company"
- Any statement that associates a fact with a person's name

**General fact patterns:**
- Industry insights
- Technical learnings
- Process improvements

### 2. Process Person Facts

For each person fact:

1. **Fuzzy match** the name against `Vault/4 - CRM/contacts.yaml`
   - If confidence >= 85%: Use matched contact name
   - If confidence 60-85%: Ask user to confirm
   - If no match: Create new contact

2. **Update profile file** at `Vault/4 - CRM/Profiles/{name}.md`:
   - If file exists: Append to Notes section
   - If not exists: Create new profile

3. **Update contacts.yaml** if new contact created

### 3. Process General Facts (Optional)

If there are non-person general learnings:
- Append to `Vault/Core Memory/Learnings.md`
- Format: `- [YYYY-MM-DD] {fact}`

### 4. Report Results

Output summary:
```
Memory Updated:
- Alex Chen: Added 2 facts to profile
- [NEW] Sam Lee: Created profile with 1 fact
- Core Memory: Added 1 general learning
```

## Profile File Format

**Location:** `Vault/4 - CRM/Profiles/{name}.md`

```markdown
# {Name}

## Quick Facts
- Company: {company}
- Relationship: {level}
- Last contact: {date}

## Notes
- {fact 1}
- {fact 2}

## Interaction Log
### {date} - {source}
- {details}
```

## Example Usage

**Conversation:**
> User: "Just had a call with Alex. He moved to NewFund from Acme Ventures. He mentioned they just closed a new fund."

**Run:** `/memory`

**Result:**
```
Memory Updated:
- Alex Chen:
  + Now at NewFund (moved from Acme Ventures)
  + Recently closed a new fund
```

**Profile updated:** `Vault/4 - CRM/Profiles/Alex Chen.md`

## Action Items (next_step)

Action items are stored in `contacts.yaml` as the `next_step` field per contact. The CRM Dashboard is auto-generated from this data.

**Source of truth:** `Vault/4 - CRM/contacts.yaml`
**Display:** `Vault/4 - CRM/CRM Dashboard.md` (regenerated by Telegram agent)

### Setting Action Items

When conversation mentions a future action for a person, set their `next_step`:

**Trigger patterns:**
- "X needs to do Y"
- "Remind me to follow up with X about Y"
- "Need to send X the proposal"
- "Schedule call with X next week"
- "X: action item is Y"

**Example:**
> "Alex needs to join the product call tomorrow"

**Action:** Set `next_step = "Join product call"` for Alex in contacts.yaml

### Completing Action Items

When conversation indicates an action is **completed**, clear the `next_step`:

**Completion signals:**
- "Done with X" / "Completed X" / "Finished X"
- "Had the call with X" / "Met with X"
- "Sent X the message" / "Followed up with X"
- Action explicitly marked as done

**Example:**
> "Just finished the product call with Alex. Went well."

**Action:** Set `next_step = null` for Alex in contacts.yaml

### Manual Commands

User can explicitly say:
- "Set action item for Alex: review the proposal"
- "Remove Alex's action item"
- "Mark Alex's follow-up as done"
- "Clear action item for [name]"

### Input Format

```typescript
interface MemoryInput {
  personFacts: { name: string; fact: string }[];
  generalFacts: string[];
  completedActions?: string[];           // Names to clear next_step
  setNextSteps?: { name: string; action: string }[];  // Names to set next_step
  source?: string;
}
```

## Notes

- This skill complements the Granola auto-processing
- Granola captures call summaries automatically
- /memory captures ad-hoc learnings from conversations
- Both write to the same profile files
- Action items are stored in contacts.yaml (single source of truth)
- CRM Dashboard is display-only (regenerated from contacts.yaml)
